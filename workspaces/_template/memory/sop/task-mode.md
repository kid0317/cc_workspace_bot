# SOP：🎯 任务模式执行规范

**适用场景**：用户要求完成一项具体、有明确产出的任务（调研、整理、分析、生成报告等）。
**与其他模式的边界**：单次问答 / 闲聊 / 单条数据查询 / 无交付物的开放讨论 → 不走本 SOP，交由默认对话或自定义模式处理。

---

## 阶段 0：识别任务模式触发

**触发信号**：
- 「帮我调研…」「帮我整理…」「帮我分析并出报告…」
- 「我想了解 X，给我一份报告」
- 任何需要多步骤、有明确交付物的任务

**不触发**：
- 单次响应可完结的问答（→ 默认对话 / 自定义模式）
- 开放性观点探讨且用户未要求交付物（→ 默认对话 / 自定义讨论模式）
- 仅需要"快速看一眼"的数据查询（→ 默认对话）

---

## 阶段 1：需求澄清（禁止跳过）

**步骤**：
1. 理解任务意图：目的是什么？受众是谁？产出形态？
2. 创建任务目录和需求文档：

```
mission/YYYYMMDD_<任务名>/
  brief.md        ← 需求文档（本阶段产出）
  result.md       ← 执行结果（阶段 3 产出，本阶段禁止生成）
  [其他中间文件]
```

3. `brief.md` 结构：

```markdown
# 任务需求文档

## 任务理解
[对用户任务的理解与拆解]

## 执行方案
[打算怎么做，分几步，用哪些工具/数据源/sub-agent]

## 待确认项
- [ ] 问题 1
- [ ] 问题 2

## 用户原始需求
[原文引用]
```

4. 向用户反问待确认项，明确告知：「**以上是我的理解和执行方案，确认后我开始执行**」

---

## 阶段 2：确认启动

用户回复「开始」/「确认」/「可以」/「执行」或明确同意后进入执行。
若用户有修正，更新 `brief.md` 的待确认项，再次确认后启动。

---

## 阶段 3：任务执行

**原则**：
- 根据任务复杂度拆解子任务，调用 sub-agent 并行执行保证深度
- 数据收集、分析、撰写可并行的尽量并行
- 所有中间文件存入 `mission/<任务名>/`
- 执行过程中若遇重大不确定，暂停告知用户

**sub-agent 拆解参考**：
- 数据收集 agent（外部数据源 / 搜索 / 抓取）
- 分析 agent（逻辑推理、比较）
- 撰写 agent（结构化输出）

---

## 阶段 4：交付

1. 生成 `result.md`，结构：

```markdown
# [任务名] 执行报告

## 执行概述
- 任务目标：
- 执行时间：
- 主要方法：
- 关键发现：

---

## 正文
[完整结果内容]

---

## 附录
[数据源、参考链接等]
```

2. 上传飞书文档（调用 `feishu_ops` skill），返回文档链接给用户。
3. 在对话中输出：执行概述（3-5 句）+ 飞书链接。

---

## 阶段 5：任务归档与 SOP 更新

执行完成后：
- 若本次任务有新的方法论或踩坑，更新本 SOP 或在 `memory/sop/` 下新建专题 SOP
- 更新 `memory/MEMORY.md` 中的相关记录（如有持久价值）

---

## 禁止事项

- 禁止跳过需求澄清阶段直接执行
- 禁止在阶段 1 产出 `result.md`（必须先确认）
- 禁止遗漏飞书上传步骤
- 禁止将 `brief.md` 和 `result.md` 混用同一文件
- 禁止在用户未要求交付物的开放讨论中强行进入任务模式
