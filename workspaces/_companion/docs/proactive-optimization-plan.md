**日期**：2026-04-12  
**状态**：已实施

## 问题一：主动唤醒频率与自然感

### 根因分析

当前 `task_templates/proactive_reach.yaml` 注释写的是"每天 10:00 和 20:00"，实际 cron `*/37 8-23 * * *` 虽然每37分钟触发一次，但 `SKILL.md` 中的 **Step 2.5 概率机制**导致实际发送极少：

```
SEND_INTENT_SCORE = 15 + MOOD_DELTA + STATUS_DELTA
# 基础15%，调整后约15~55%，中位约25%
```

但 cron 触发18次/天（8-23点，每37分钟），而跳过上限是4次才强制发。  
问题在于：**两次触发之间间隔37分钟，但检查"距上次对话<2小时则不发"会过滤掉大多数触发**。  
用户刚聊完，之后2小时内所有触发都被 Step 1 过滤，导致每天实际检查机会很少。

另一个问题：整点时间触发（如 `0 10,20 * * *`）完全没有真人感。

### 设计方案

**核心思路**：提高触发频率（每小时检查），用随机微延迟打破钟点感，概率收敛到1-2次/天。

#### 1. Cron 策略

```
旧：*/37 8-23 * * *（每37分钟，约18次/天）
新：23 8-22 * * *（每小时 :23 分触发，约15次/天）
```

选 `:23` 而非 `:00` 或 `:30`，视觉上不像整点/半点触发。

#### 2. 关于随机延迟（已放弃）

~~原方案：SKILL.md 新增 `sleep $((RANDOM % 720))`~~

**放弃原因（review 发现）**：框架 `timeout_minutes: 5`，最长 sleep 12 分钟会导致 58% 概率任务被 kill，必然失败。

**替代思路**：用户感知到的不规则感来自"不知道什么时候会收到消息"，而不是"知道整点但偏了几分钟"。概率跳过机制本身已经制造了这种不确定感，每次触发 85% 静默，用户体验到的到达时间天然随机，无需额外 sleep。

#### 3. 概率调优（review 后修订）

| 参数 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| 基础发送率 | 15% | 15%（保持） | 不变 |
| 连续跳过上限 | 4次 | **8次** | 关键调整，见下方数学说明 |
| 最小间隔 | 2小时 | 2小时 | 保持不变 |
| clamp 范围 | 无 | [5%, 70%] | 防止情绪叠加后越界 |

**数学说明**（review 发现原方案漏算强制发送）：

强制发送机制：连续跳过 N 次后第 N+1 次强制，每 N+1 次触发中**至少有 1 次**是强制发送。

以每天 15 次触发为例：
- skip_count 上限 = 8 → 每 9 次触发强制 1 次
- 强制发送：floor(15/9) ≈ 1 次
- 随机发送（剩余 6 次 × 15%）≈ 0.9 次
- **期望总发送 ≈ 1.9 次/天** ✓

旧设计（skip 上限 4）：floor(15/5) = 3 次强制 + ~0.6 随机 ≈ 3.6 次/天（过频）。

## 问题二：主动触达与日常对话的文风差异

### 根因分析

**Hook 机制只在 `UserPromptSubmit` 时触发**，定时任务执行时**完全不经过 Hook**：

```
日常对话路径：
  inject_history.py (hook) → reply_checklist.sh (hook) → CLAUDE.md → 角色回复
  
主动触达路径：
  proactive/SKILL.md → 直接生成消息
```

因此：
1. `reply_checklist.sh` 的 AI腔检测规则（A区/B区/C区/D区）**不注入到主动触达的 Claude 上下文**
2. `proactive/SKILL.md` 未显式加载 `memory/persona.md`，角色声音缺乏具体锚定
3. Step 3 的风格要求只有3条，远少于 CLAUDE.md 的完整说话规则

### 设计方案（review 后修订）

**根因补充（review 发现）**：
- CLAUDE.md 通过目录遍历**已经加载**（cmd.Dir = sessions/<id>/，向上能找到 CLAUDE.md）
- 作者框架、禁止句式、角色设定块均在上下文中
- persona.md 无需额外加载（内容已追加到 CLAUDE.md 末尾【角色设定】块）

**实际差距（两处）**：
1. `reply_checklist.sh` 是 UserPromptSubmit Hook，定时任务不触发，等效于"检察官离场"
2. `inject_history.py` 不刷新，RECENT_HISTORY.md 是旧快照，影响情绪基线感知

**修复方案**：

在 SKILL.md Step 5（消息生成）中：
- 内联 reply_checklist.sh 的核心禁止规则（A/B/C 三区）
- 明确要求"先自检，违反则重写，不向用户解释"（自检-重写指令是关键，光列规则不够）
- 新增 Step 4：读取 RECENT_HISTORY.md 获取情绪基线，作为生成语气参考

## 问题三：日常对话时间流逝感缺失

### 根因分析

**`inject_history.py` 的 `parse_timestamp()` 函数只返回日期，丢失了时间**：

```python
# 旧代码：
return datetime.strptime(s[:len(fmt)], fmt).strftime("%Y-%m-%d")  # 只保留日期

# 写入 RECENT_HISTORY.md 的格式：
**用户**（2026-04-12）：[内容]
**角色**（2026-04-12）：[内容]
```

下午2点和晚上8点的消息，时间戳都是 `2026-04-12`，Claude 无法判断"距上次聊天已过6小时"。

CLAUDE.md 启动流程第3步明确说：
> "读取最后一条消息的时间戳，与 SESSION_CONTEXT.md 中 Current datetime 对比，计算间隔"

但传入的时间戳精度只有天，这个指令实际上无法执行。

### 设计方案

**修复 `inject_history.py`**：`parse_timestamp()` 返回 `YYYY-MM-DD HH:MM`。

```python
# 新代码：
return datetime.strptime(...).strftime("%Y-%m-%d %H:%M")  # 保留小时和分钟
```

写入格式变为：
```
**用户**（2026-04-12 14:23）：[内容]
**角色**（2026-04-12 14:31）：[内容]
```

Claude 读到"最后一条消息 14:31"，当前时间 20:15，间隔 5h44m，触发"2-6小时"行为规则：
> "时间感明显，不主动接续之前话题；可自然提起时间段（下午、傍晚）"

**同时补强 CLAUDE.md 时间感知的实现要点**，明确写出从 RECENT_HISTORY.md 提取时间的方法，避免歧义。
